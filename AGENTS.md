# AGENTS.md

## 最重要ルール

- 思考は日本語で行うこと。
- 回答は日本語で行うこと。
- 生成物は日本語で作成すること。
- アーティファクト内部の記述も原則として日本語で統一すること。
- フィードバック、説明、進捗報告も日本語で行うこと。

## 開発ルール

### リスク管理ゲートの実行を必須とする

- 新機能追加、バグ修正、リファクタリング、AI生成コードを含むすべての変更は、コミット前に必ずリスク管理ゲートを実行すること。
- このリポジトリの標準ゲートは `npm run risk:gate` とする。
- リポジトリ全体への影響がある変更、依存関係変更、CI変更、hook変更、セキュリティ関連変更では `npm run risk:gate:all` も実行すること。
- ゲート結果が `FAIL` の場合、問題を解消するまでコミットおよびプッシュを禁止する。
- ゲート結果が `WARN` の場合、人間による内容確認と解消判断を行うまでコミットおよびプッシュを行わないこと。
- リスク管理ゲートを実質的にクリアした変更のみ、コミットおよびプッシュを許可する。

### コミット・プッシュ手順

すべての変更で、以下の順序を守ること。

```bash
git diff
npm run risk:gate
git add ...
git commit -m "..."
git push
```

追加ルール:

- リスク管理ゲート未実行のコードはコミット・プッシュしてはならない。
- `pre-push` hook は常に有効化し、プッシュ前にリスク管理ゲートを再実行すること。
- hook が外れている場合は `npm run hooks:install` で再設定すること。
- CI でも同じリスク管理ゲートを継続して実行すること。

### 適用対象

このルールは以下すべてに適用する。

- AI生成コード
- 手動修正コード
- 設定ファイル変更
- workflow / CI 変更
- セキュリティ関連変更

## エラー・検証

- コード実装完了後は、利用中のすべてのコンテナでエラー確認を行うこと。
- エラーが見つかった場合は、修正して再度確認すること。
- コミットおよびプッシュは、必要な検証を終えてから行うこと。

### 動作確認の方針

- 軽微な変更では、Codex 側は実装、デバッグ、リスク管理ゲート実行までを担当し、ブラウザ操作を含む最終的な動作確認は原則としてユーザーが行う。
- 軽微な変更の例は、色合いや見た目の修正、表記ラベルの修正、小機能の追加・修正などとする。
- Playwright などを用いた総合的な動作確認は、明確に必要な場合や高リスク変更に限定して実施する。
- 動作確認をこちらで実施するかどうかの判断に迷う場合は、先にユーザーへ確認し、その指示に従うこと。

### 低消費での依頼フロー

- レート使用量を抑えることを優先する場合、まず対象ファイルの特定だけを依頼し、その後にユーザーが対象ファイルを指定して本依頼する運用を基本とする。
- 対象ファイルが不明な場合の初回依頼では、必要最小限の調査だけを行い、原因や対象ファイルの特定までに留めること。
- 対象ファイルが明確な軽微修正では、最初からファイル指定で依頼する運用を推奨する。
- 不具合調査では、初回はざっくりした依頼でもよいが、実装前に原因と対象ファイルを整理してから着手すること。
- Codex は、ユーザーから `AGENTS.md の低消費方針で` と指定された場合、この方針を優先して、確認範囲・変更範囲・検証範囲を必要最小限に絞ること。

### 低消費用の短縮命令

- ユーザーが `低消費-調査` と指示した場合は、AGENTS.md の低消費方針を適用し、必要最小限の調査だけを行うこと。
- `低消費-調査` では、対象ファイルや原因候補の特定までに留め、明示的な追加指示がない限り実装は行わないこと。
- ユーザーが `低消費-実装` と指示した場合は、AGENTS.md の低消費方針を適用し、指定された対象ファイルだけを前提に実装すること。
- `低消費-実装` では、明示的な追加指示がない限り、検証はデバッグと `npm run risk:gate` までに留め、Playwright などの総合動作確認は行わないこと。
- `低消費-実装` で対象ファイル指定がない、または不足している場合は、すぐ実装に入らず、必要最小限の確認だけで対象ファイル候補を提示すること。

## ExecPlan

- 複雑な機能追加、大きな設計変更、大規模なリファクタリングでは、設計から実装までの ExecPlan を使用すること。
- 実装前に、作業の段取り、影響範囲、検証方針が分かる形で整理すること。

## 安全制御

- 破壊的操作を禁止する。例: `rm`, `rmdir`, `del`, `format`
- `migrate:fresh` など、DB を破壊する操作を禁止する。
- ディレクトリ全削除を禁止する。
- `.env` や秘密情報を出力してはならない。
- 変更は対象リポジトリ配下に限定すること。

### 参照コマンド

```bash
# ローカルの変更差分を対象に実行
npm run risk:gate

# リポジトリ全体を対象に実行
npm run risk:gate:all

# Git hook を再設定
npm run hooks:install
```
