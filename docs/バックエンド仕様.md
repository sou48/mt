# MT バックエンド仕様

## 目的

本書は、MT のモック実装を実運用向けに置き換えるための、初期バックエンド仕様をまとめたものです。対象は以下です。

- DB 設計の初期方針
- 初期実装に含める REST API
- 後続フェーズへ送る未実装項目の整理

## スコープ

初期実装の対象は以下です。

- 認証
- 会社
- 案件
- メッセージ
- 署名
- 辞書
- 添付ファイル
- 検索

初期実装の対象外です。

- 履歴参照 API
- 管理者向け詳細運用 API
- 実送信機能
- AI / 翻訳のサーバー中継実装

## 基本方針

- ユーザーは社内ユーザー単位でログインする
- ユーザーは 1 社にのみ所属する
- 案件は必ず 1 社に属する
- 一般ユーザーは、自分が担当している案件だけ閲覧できる
- 管理者は全会社、全案件、全メッセージを閲覧できる
- メッセージは必ず案件に属する
- ただし案件未確定の受信は、会社ごとの未分類仮案件へ保存する
- メッセージは 1 つの `messages` テーブルで管理する
- 返信は「MT 内で作成・管理する文面」を指し、実送信は責務に含めない
- API は REST で実装する
- 認証はセッション認証を採用する

## 用語

- 受信: 外部メールやチャットから MT に取り込んだメッセージ
- 返信: 相手へ返すために MT 内で管理する文面
- 下書き: 返信の途中状態
- 未分類仮案件: 案件未確定の受信を一時保存するための特別案件

## 権限

### ユーザー種別

- 一般ユーザー
- 管理者

### 一般ユーザー

- 自分が担当する案件だけ見える
- 担当案件を持つ会社だけ会社一覧に見える
- 会社作成、案件作成ができる
- システム共通辞書を編集できる
- 自社の会社共通辞書を編集できる
- 自分の署名だけ編集できる

### 管理者

- 全会社、全案件、全メッセージを閲覧できる
- 全ユーザーを管理できる

## テーブル設計案

### 1. companies

会社マスタです。

| カラム | 型の想定 | 備考 |
| --- | --- | --- |
| id | bigint | PK |
| name | varchar | 会社名 |
| created_at | timestamp | 作成日時 |
| updated_at | timestamp | 更新日時 |
| deleted_at | timestamp nullable | 論理削除 |

ルール:

- 内部 ID と会社名で管理する
- 会社名は完全一致なら作成不可
- 類似名は警告のみ

### 2. users

ユーザーマスタです。

| カラム | 型の想定 | 備考 |
| --- | --- | --- |
| id | bigint | PK |
| company_id | bigint | FK -> companies.id |
| email | varchar | ログイン ID、一意 |
| display_name | varchar | 表示名 |
| role | varchar | `user` / `admin` |
| password_hash | varchar | パスワードハッシュ |
| created_at | timestamp | 作成日時 |
| updated_at | timestamp | 更新日時 |
| deleted_at | timestamp nullable | 論理削除 |

### 3. projects

案件マスタです。

| カラム | 型の想定 | 備考 |
| --- | --- | --- |
| id | bigint | PK |
| company_id | bigint | FK -> companies.id |
| name | varchar | 案件名 |
| is_unclassified | boolean | 未分類仮案件かどうか |
| created_by_user_id | bigint nullable | 作成者 |
| created_at | timestamp | 作成日時 |
| updated_at | timestamp | 更新日時 |
| deleted_at | timestamp nullable | 論理削除 |

ルール:

- 内部 ID と案件名で管理する
- 同一会社内で完全一致案件名は作成不可
- 類似名は警告のみ
- 会社ごとに未分類仮案件を 1 つ持つ
- 未分類仮案件は通常一覧に表示するが、未分類と明示する
- 未分類仮案件は名前変更、削除不可
- 案件状態は持たない

### 4. user_projects

案件担当者の中間テーブルです。

| カラム | 型の想定 | 備考 |
| --- | --- | --- |
| id | bigint | PK |
| user_id | bigint | FK -> users.id |
| project_id | bigint | FK -> projects.id |
| created_at | timestamp | 作成日時 |
| updated_at | timestamp | 更新日時 |
| deleted_at | timestamp nullable | 論理削除 |

ルール:

- 権限ロールは持たない
- 担当しているかどうかだけを管理する

### 5. messages

受信、返信、下書きを一元管理するテーブルです。

| カラム | 型の想定 | 備考 |
| --- | --- | --- |
| id | bigint | PK |
| project_id | bigint | FK -> projects.id |
| message_type | varchar | `received` / `reply` / `draft` |
| channel_type | varchar | `email` / `chat` / `phone_memo` / `other` |
| subject | varchar nullable | メール系のみ |
| source_sender_name | varchar nullable | 元送信者名 |
| source_sender_address_or_account | varchar nullable | 送信元情報 |
| source_sent_at | timestamp nullable | 元メッセージ日時 |
| registered_by_user_id | bigint nullable | 受信登録者 |
| created_by_user_id | bigint nullable | 返信や下書きの作成者 |
| source_text | text nullable | 取り込んだ原文 |
| source_language | varchar nullable | 原文言語 |
| translated_text | text nullable | 翻訳文 |
| translated_language | varchar nullable | 翻訳文言語 |
| japanese_text | text nullable | 日本語文 |
| partner_text | text nullable | 相手言語文 |
| language_pair | varchar nullable | 例: `日本語<>韓国語` |
| signature_id | bigint nullable | 使用署名 ID |
| signature_snapshot | text nullable | 当時の署名本文 |
| created_at | timestamp | 作成日時 |
| updated_at | timestamp | 更新日時 |
| deleted_at | timestamp nullable | 論理削除 |

ルール:

- 返信と下書きは、日本語文と相手言語文の両方を正本として保存する
- 件名はメール系チャネルのみ使用する
- 並び順は `source_sent_at` 優先、なければ `created_at`
- 返信、下書きの更新は最新を本体に持ち、更新前を履歴へ退避する

### 6. attachments

添付ファイルテーブルです。

| カラム | 型の想定 | 備考 |
| --- | --- | --- |
| id | bigint | PK |
| message_id | bigint | FK -> messages.id |
| original_name | varchar | 元ファイル名 |
| stored_path | varchar | 保存先パス |
| mime_type | varchar | MIME type |
| file_size | bigint | サイズ |
| uploaded_by_user_id | bigint nullable | 登録者 |
| created_at | timestamp | 作成日時 |
| updated_at | timestamp | 更新日時 |
| deleted_at | timestamp nullable | 論理削除 |

ルール:

- 初期実装では VPS ローカルディスクへ保存する
- ダウンロードできるのは、その案件を閲覧できるユーザーだけ

### 7. signatures

ユーザー署名テーブルです。

| カラム | 型の想定 | 備考 |
| --- | --- | --- |
| id | bigint | PK |
| user_id | bigint | FK -> users.id |
| name | varchar | 署名名 |
| japanese_text | text | 日本語署名 |
| partner_text | text nullable | 相手言語署名 |
| is_default | boolean | デフォルト署名 |
| created_at | timestamp | 作成日時 |
| updated_at | timestamp | 更新日時 |
| deleted_at | timestamp nullable | 論理削除 |

ルール:

- 署名はユーザー単位で管理する
- 自分の署名だけ編集できる

### 8. dictionary_entries

辞書エントリテーブルです。

| カラム | 型の想定 | 備考 |
| --- | --- | --- |
| id | bigint | PK |
| scope_type | varchar | `system` / `company` |
| company_id | bigint nullable | 会社共通辞書のときのみ使用 |
| source_term | varchar | 原文 |
| target_term | varchar | 訳語 |
| note | text nullable | メモ |
| language_pair | varchar | 例: `日本語<>韓国語` |
| created_by_user_id | bigint nullable | 作成者 |
| updated_by_user_id | bigint nullable | 更新者 |
| created_at | timestamp | 作成日時 |
| updated_at | timestamp | 更新日時 |
| deleted_at | timestamp nullable | 論理削除 |

ルール:

- 辞書はシステム共通辞書と会社共通辞書の 2 階層
- 案件別辞書は持たない
- システム共通辞書は全ユーザーが編集できる
- 会社共通辞書はその会社のユーザーなら全員編集できる

### 9. password_reset_tokens

パスワードリセット用トークンです。

| カラム | 型の想定 | 備考 |
| --- | --- | --- |
| id | bigint | PK |
| user_id | bigint | FK -> users.id |
| token_hash | varchar | トークンハッシュ |
| expires_at | timestamp | 有効期限 |
| used_at | timestamp nullable | 使用日時 |
| created_at | timestamp | 作成日時 |

ルール:

- 有効期限は 30 分
- 送信元メールアドレスはシステム共通

## 履歴テーブル方針

主要データは更新前スナップショットを履歴テーブルへ保存し、履歴参照 API から参照できる前提で設計する。

対象:

- messages
- signatures
- dictionary_entries

想定テーブル:

- message_histories
- signature_histories
- dictionary_entry_histories

共通方針:

- 更新前の内容を保存する
- 誰が更新したかを保存する
- 変更理由は初期実装では任意

## 初期 REST API 一覧

### 認証

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`
- `POST /api/auth/password-reset/request`
- `POST /api/auth/password-reset/confirm`

### 会社

- `GET /api/companies`
- `POST /api/companies`
- `GET /api/companies/:companyId`
- `PATCH /api/companies/:companyId`

補足:

- 一般ユーザーには、自分が担当案件を持つ会社だけ返す

### 案件

- `GET /api/projects`
- `POST /api/projects`
- `GET /api/projects/:projectId`
- `PATCH /api/projects/:projectId`
- `POST /api/projects/:projectId/assign-users`
- `POST /api/projects/:projectId/reclassify-messages`

補足:

- 一般ユーザーには自分の担当案件だけ返す
- 未分類仮案件は通常一覧に含める

### メッセージ

- `GET /api/messages`
- `GET /api/messages/:messageId/histories`
- `POST /api/messages/received`
- `POST /api/messages/replies`
- `PATCH /api/messages/:messageId`
- `DELETE /api/messages/:messageId`
- `POST /api/messages/:messageId/move-project`

補足:

- `POST /api/messages/received`
  - 受信取り込み用
- `POST /api/messages/replies`
  - 返信または下書き作成用
- `PATCH /api/messages/:messageId`
  - 下書きや返信の更新
- `POST /api/messages/:messageId/move-project`
  - 未分類仮案件から正式案件への振り替えに使用

### 添付ファイル

- `POST /api/messages/:messageId/attachments`
- `GET /api/messages/:messageId/attachments`
- `GET /api/attachments/:attachmentId/download`
- `DELETE /api/attachments/:attachmentId`

### 署名

- `GET /api/signatures`
- `GET /api/signatures/:signatureId/histories`
- `POST /api/signatures`
- `PATCH /api/signatures/:signatureId`
- `DELETE /api/signatures/:signatureId`

### 辞書

- `GET /api/dictionaries/system`
- `GET /api/dictionaries/system/:entryId/histories`
- `POST /api/dictionaries/system`
- `PATCH /api/dictionaries/system/:entryId`
- `DELETE /api/dictionaries/system/:entryId`
- `GET /api/companies/:companyId/dictionaries`
- `GET /api/companies/:companyId/dictionaries/:entryId/histories`
- `POST /api/companies/:companyId/dictionaries`
- `PATCH /api/companies/:companyId/dictionaries/:entryId`
- `DELETE /api/companies/:companyId/dictionaries/:entryId`

### 検索

- `GET /api/search/companies`
- `GET /api/search/projects`
- `GET /api/search/messages`

補足:

- メッセージ検索対象:
  - 原文
  - 翻訳文
  - 件名
  - 送信者名
  - 送信元情報

## API 共通方針

- 一覧 API はすべてページネーション対応
- メッセージ UI はスクロール表示だが、裏側は分割取得する
- 論理削除を採用する
- 検索条件は権限で絞り込んだ上で適用する
- 添付ファイルダウンロードも権限チェック必須

## バリデーション方針

- 会社名
  - 完全一致不可
  - 類似名警告
- 案件名
  - 同一会社内で完全一致不可
  - 類似名警告
- ユーザーメールアドレス
  - 一意制約
- 未分類仮案件
  - 名前変更、削除不可
