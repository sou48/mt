# ExecPlan: 初期バックエンド実装

## 目的

MT のモック実装を、初期バックエンド付きの実運用前提構成へ段階移行する。

完了条件:

- PostgreSQL ベースの永続化が導入されている
- セッション認証が動作する
- 認証、会社、案件、メッセージ、署名、辞書、添付、検索 API の初期版が動作する
- 現行フロントの主要画面が API を使って動作する土台ができる

## 前提

- 本計画は [バックエンド仕様](/home/takeshi/multi-translate/docs/バックエンド仕様.md)、[テーブル定義確定案](/home/takeshi/multi-translate/docs/テーブル定義確定案.md)、[技術選定](/home/takeshi/multi-translate/docs/技術選定.md) を前提とする
- 実装対象は MT リポジトリ内に限定する
- 各実装ステップの前後でリスク管理ゲートを実行する

## 実装順

### フェーズ0: 土台準備

目的:

- バックエンド実装を置ける最小構成を用意する

作業:

- Express のディレクトリ構成を整理する
- API ルーティング層を作る
- 環境変数の読み込み方法を決める
- DB 接続基盤を導入する
- Docker Compose に PostgreSQL を組み込む

成果物:

- `server.js` から API ルーティングへ委譲できる構成
- PostgreSQL コンテナ起動
- DB 接続確認

検証:

- API のヘルスチェック
- DB 接続確認

### フェーズ1: 認証

目的:

- ログイン状態をサーバー管理に移す

作業:

- `users` テーブル実装
- 管理者、一般ユーザーの初期シード方針を決める
- セッション認証導入
- ログイン、ログアウト、`me` API 実装
- パスワードリセットトークンとメール送信の実装

成果物:

- ログイン API
- セッション Cookie
- パスワードリセット API

検証:

- 正常ログイン
- 不正ログイン
- ログアウト
- トークン期限切れ

### フェーズ2: 会社 / 案件

目的:

- 一般ユーザーの閲覧範囲と案件単位権限を成立させる

作業:

- `companies`, `projects`, `user_projects` 実装
- 会社、案件の一覧、詳細、作成 API 実装
- 完全一致重複チェック
- 類似名警告ロジック
- 会社作成時の未分類仮案件自動作成

成果物:

- 会社 API
- 案件 API
- 担当紐付け基盤

検証:

- 一般ユーザーは担当案件のみ閲覧できる
- 管理者は全件閲覧できる
- 重複チェックが働く

### フェーズ3: メッセージ / 添付

目的:

- 受信、返信、下書きの実データ管理を開始する

作業:

- `messages`, `attachments` 実装
- 受信取り込み API
- 返信 / 下書き作成 API
- メッセージ更新 API
- 未分類仮案件からの振り替え API
- 添付アップロード、一覧、ダウンロード API

成果物:

- メッセージ API
- 添付 API

検証:

- 受信メッセージ登録
- 返信作成
- 下書き更新
- 添付アップロード
- 権限外ダウンロード拒否

### フェーズ4: 署名 / 辞書

目的:

- 設定系データを API 経由へ移す

作業:

- `signatures`, `dictionary_entries` 実装
- 署名 CRUD
- システム共通辞書 CRUD
- 会社共通辞書 CRUD
- 編集者記録

成果物:

- 署名 API
- 辞書 API

検証:

- 自分の署名だけ編集可能
- 会社辞書は自社ユーザーだけ編集可能
- システム共通辞書は全ユーザー編集可能

### フェーズ5: 検索 / ページネーション

目的:

- 一覧系の実用性を確保する

作業:

- 会社検索
- 案件検索
- メッセージ検索
- 一覧 API 共通ページネーション

成果物:

- 検索 API
- 一覧 API の共通返却形式

検証:

- 権限範囲内のみヒットする
- 件名、送信者名、送信元情報まで検索できる

### フェーズ6: フロント接続の段階移行

目的:

- `localStorage` 依存を減らし、API 利用へ移す

作業:

- 会社一覧の API 接続化
- 案件一覧の API 接続化
- メッセージ一覧の API 接続化
- 署名画面の API 接続化
- 辞書画面の API 接続化

成果物:

- API 利用に置き換わった主要画面

検証:

- ログイン後に実データが見える
- 受信 / 返信 / 下書きが永続化される

## 優先実装順

最短で価値が出る順:

1. DB 基盤
2. 認証
3. 会社 / 案件
4. メッセージ / 添付
5. 署名 / 辞書
6. 検索
7. フロント接続

## リスク

- 現在のフロントは `localStorage` 前提のため、全面一括置換は壊れやすい
- 添付保存をローカル固定にするため、将来移設時の考慮が必要
- システム共通辞書を全ユーザー編集可にしているため、履歴と監査が重要

## 実装単位ごとの完了判定

- 各フェーズ完了時に `npm run risk:gate` を実行する
- 依存関係、CI、認証、セキュリティ関連変更では `npm run risk:gate:all` を実行する
- 可能な限りコンテナ上で動作確認する
- 実装後は [タスク管理](/home/takeshi/multi-translate/docs/タスク管理.md) の該当項目を更新する
